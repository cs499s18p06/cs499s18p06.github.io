Week 10: April 15, 2018

At this point the main menu scene was functioning correctly. I figured out how to traverse in and out of scenes without any problems. From here, I worked on getting the scene into a unique prefab and then adding the scene into the game. Its set up in the build function for us to start in the main menu scene and from there go into the game.

Next step was implementing my pause functionality which did not work as planned. Google Cardboard and Unity’s User Interface Functionality does not work properly. In a normal game where you can use a mouse, most of Unity3D’s interface design, is in screen overlay mode. This means that no matter where the character is in the game, if they pause or need to use a menu, the UI will be right in front of the characters view. Unfortunately, Google Cardboard requires me to use world space camera overlay. Since our game is an on rails shooter, the user is actively traveling around the map, which mean a world space UI overlay would need a script to actively put itself in front of the user. This is a script I would still need to work on.

Otherwise, when you look down in your mine cart, you see an animation occur and if you let the animation run down it quits the game and takes you back to the main menu screen.

Next on the list is to take the score mechanic and print a score number at the end of the race to showcase the amount of ghost they killed.

Week 9: April 8, 2018

This week was a large focus on getting the project working. We had our ideas implemented but no real progress on putting the game together. With problems arising in certain situations, I had to figure out a bit more on my part of the project. I spent the majority of my time in the early part of the week learning about Google Cardboard functionality. In order to test the main menu screens, I also needed to have a working cursor and reticle.

In an earlier meeting, we decided on using the newest version of google cardboard which incorporated a button on the side of the headset. We agreed to use this and work towards making a game that used the on click functionality. However, I did some research on things like GazeClick which was easy to implement and helped with some User Interface design.

I was able to incorporate both functions. Not only are you able to click on the button and it activate but also wait two seconds for the GazeClick animation. Basically, I created a cube and added a cube interaction function. Google cardboard has a lot of nice prefabs to help get you started in the right direction. One of those prefabs is an event system that helps determine whether or not the cursor is has entered inside of the cube or outside of the cube. From there I created a C# script that was able to take these events and find different situations for them.

When you gaze your cursor inside of the cube it will automatically start the timer for the interaction. From here I was able to create another cube that acted as the animation and created basic loading screen. After the two seconds inside of the cube or when the user clicks, it goes to the function that is supposed to switch scenes. However, loading different scenes has been very problematic when testing and I’m still figuring that out. If the user stops hovering over the cube, the animation stops and the timer goes back to zero.

Since the main menu is working properly, implementing a pause function in the game won’t be too difficult since most of the mechanics will be using the same kind of code. The only problem now is figuring out how to effectively switch in between scenes which is becoming more of a hassle than intended. The code is very simple but it comes with many problems; cameras are glitching out, not starting the game in the proper hierarchy, etc. These are things that should be sorted out as soon as possible.

Week 7: March 25, 2018

For this week, I researched more into user interface design for virtual reality. I researched into SteamVR and VRKT plugins that are provided through the Unity3D asset store. Both plugins have many great options for virtual reality implementation. Unfortunately, most of these plugins caused many problems, and also they were designed for actual virtual reality implementation and not google cardboard. As a result, I’ve had to disband from these plugins.

I’ve also researched more into the idea of how the main menu screen should be implemented. There was two scenarios that could be done. The first one is that the game recognizes where you’re facing and places the main menu screen in front of you. The other idea was to create some kind of immersive environment and the main menu could be placed on a wall or such.

Both ideas were favorable ideas, but after our meeting this week, we decided that the design implementation should be as simple as possible. Also the idea of ‘creating space’ was taken into consideration. When you put the headset on, being placed in darkness creates a sense of claustrophobia, and so some design implementation to create that ‘space’ in order for the user to have better experience is recommended.

After our meeting, we’ve also changed ideas on how we’re implementing our game. One of the biggest results that affect me is that our headset will require a button. As such, this eliminates the implementation of creating special virtual reality buttons. Beforehand, I was trying to create a way for the user to hover over a button, and after a certain amount of time, the button would be registered and the action would soon follow.

However, since we’re requiring a button for this game to preform, regular clicking should be suffice in order to create the user interface.

Other ideas for user interface, creating a pause button on the bottom of the mine cart that lets the user be able to pause the game when they request but also not interfere with the game itself. Other ideas is to keep point systems in the top corner of the screen to showcase the players score.
